About the EventBus

The Event Bus is a single-process publish/subscribe event routing library, with Swing extensions. The EventBus is fully-functional, with very good API documentation and test coverage (80+%). It has been deployed in many production environments, including financial, engineering and scientific applications.

The Event Bus is easy to use and yet very powerful. It requires no setup and the API is small and clear. The EventBus solves many problems, and promotes loose coupling, particularly with Swing applications.

The slides from the EventBus presentation at JavaOne 2006 demonstrate many ways to use the EventBus.

Though not Swing-specific, the EventBus has Swing support and the EventBus hopes to make Swing applications more componentized, modular and testable. The EventBus has its conceptual roots in the old InfoBus from JDK 1.1 days, but it is much simpler and smaller. The idea is similar to a computer system bus - components plug into a single bus and each can listen for information passed on it. The EventBus is different from the tightly-coupled Swing listener paradigm where listeners attach to and listen on specific components. Another difference is that the EventBus is not limited to Swing events. Any object can be published on the EventBus. In typicaly usage, you create events for your own applications, or publish objects that you already have in your application domain.

Components can subscribe to events passed on the Event Bus by Class or by String (called a topic name). Subscriptions use weak references by default, avoiding memory leaks common in swing application when listeners are not removed.

The Event Bus is related to messaging systems like JMS whereby many processes publish messages on a topic and other systems can listen on the topic. JMS implementations take care of efficiently routing the messages. The major advantage to JMS and the EventBus is that it introduces a decoupling layer. Components only have to know about the bus, not each subscriber and publisher that they may be interested in. The EventBus differs from JMS since it meant to be used by a single JVM.

Examples

A typical use case for the EventBus is a status bar. Applications can avoid coupling components to status bars (and vice versa) by publishing status events on the EventBus and having the status bar subscribe to status events. (A key advantage of the EventBus is that the contracts between components are defined by the events that are published between them.) :

StatusEvent.java 

public StatusEvent { 
   Sting statusText;
   
   public StatusEvent(String statusText) { 
      this.statusText = statusText;
   }
   public String getStatusText() {
      return statusText;
   }
}

AnySwingController.java 

...   
EventBus.publish(new StatusEvent("Looking up data..."));

StatusBar.java 

public StatusBar extends JLabel implements EventSubscriber {
   public StatusBar() {
      EventBus.subscribe(StatusEvent.class, this);
   }
   public void onEvent(Object evt) {
      StatusEvent statusEvent = (StatusEvent)evt;
	  this.setText(statusEvent.getStatusText();
   }
   }

}

With Annotations, it's a bit simpler (StatusEvent.java and AnySwingController.java stay the same):

StatusBar.java 

public StatusBar extends JLabel {
              public StatusBar() {
                 AnnotationProcessor.process(this);
              }
              @EventSubscriber(eventClass=StatusEvent.class)
              public void onEvent(StatusEvent statusEvent) {
                 this.setText(statusEvent.getStatusText();
              }
           }
